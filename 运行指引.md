# 代码查重抄袭检测系统 - 完整运行指引

## 系统概述

本系统是一个基于 **E-PDG（Effect-augmented Program Dependence Graph，带副作用的程序依赖图）** 的多语言代码抄袭检测流水线。与传统的文本比对工具不同，本系统在**图结构层面**分析代码，能够有效抵抗变量重命名、代码重构、语句重排等常见的抄袭伪装手段。

**核心技术栈**：MinHash/LSH 快速召回 + 图指纹 + 副作用签名 + DACD 模板聚类

## 快速开始：一键运行

系统提供 **两条独立的分析路径**，均可完整运行端到端流水线：

### 路径 A：AST 路径（推荐）

基于 Python AST 构建 PDG，配合 reaching definitions 不动点算法进行数据流分析。
跨 Python 版本稳定，支持 Python/C/C++/Java 多语言。

```bash
cd /path/to/code_pla

# 纯 LSH 模式（最简单）
python scripts/pipeline_pure_lsh_dacd.py --src_root data/submissions_simple

# 启用 DACD 模板聚类模式（更精确，会降低模板代码的权重）
python scripts/pipeline_pure_lsh_dacd.py --src_root data/submissions_simple --with_dacd
```

**产物位置**：
- JSON 结果：`data/artifacts/retrieval/baseline_pairs.json`
- HTML 报告：`data/artifacts/reports/baseline_pairs.html`

### 路径 B：字节码路径

基于 CPython 字节码构建真实基本块 CFG，使用 SSA 风格 worklist 数据流分析 + 后支配者控制依赖。
理论上更精确，但仅支持 Python，且对 Python 版本敏感（推荐 3.11+）。

```bash
cd /path/to/code_pla

# 纯 LSH 模式
python scripts/pipeline_bytecode.py --src_root data/submissions_simple

# 启用 DACD
python scripts/pipeline_bytecode.py --src_root data/submissions_simple --with_dacd
```

**产物位置**（与 AST 路径分开，不会互相覆盖）：
- JSON 结果：`data/artifacts/retrieval/baseline_pairs_bytecode.json`
- HTML 报告：`data/artifacts/reports/baseline_pairs_bytecode.html`

### 两条路径的对比

| 维度 | AST 路径 | 字节码路径 |
|------|---------|-----------|
| 入口脚本 | `pipeline_pure_lsh_dacd.py` | `pipeline_bytecode.py` |
| 阶段1 脚本 | `build_epdg.py` | `build_epdg_bytecode.py` |
| 核心构建器 | `epdg/ast_pdg_builder.py` | `legacy_bytecode_path/bytecode_pdg_builder.py` |
| 节点粒度 | 每条语句一个节点 | 每个基本块一个节点 |
| CFG 构建 | AST 结构递归分解（分支/合流/回边） | 字节码指令级跳转分析 |
| 数据流 | Reaching definitions 不动点迭代 | SSA 风格 worklist + 虚拟栈模拟 |
| 控制依赖 | 结构化（直接嵌套标记） | 后支配者算法（教科书标准） |
| 语言支持 | Python / C / C++ / Java | 仅 Python |
| Python 版本兼容 | 所有 3.8+ | 推荐 3.11+（字节码格式依赖） |
| 稳定性 | 高 | 中（复杂字节码模式可能触发边界情况） |
| 阶段 2-5 | 完全相同 | 完全相同 |

> **注意**：两条路径只有阶段 1（源码→E-PDG JSON）不同，阶段 2-5（多视图增强→LSH 索引→候选对→评分报告）完全共享同一套代码。

## 流水线详细步骤说明

### 阶段 1：构建 E-PDG（源码 → 图 JSON）

**脚本**：`scripts/build_epdg.py`

```bash
python scripts/build_epdg.py \
  -i data/submissions_simple \
  -o data/artifacts/json \
  --effects effect_summaries.yaml
```

**做了什么**：
1. 遍历输入目录中的所有源文件，按扩展名自动检测语言
2. 对每个 Python 文件：解析 AST → 构建控制流图(CFG) → 构建程序依赖图(PDG) → 匹配副作用签名 → 提取字节码 opcode token 序列
3. 对每个 C/C++/Java 文件：调用对应前端处理
4. 输出：每个源文件对应一个 `*.epdg.json` 文件

**输出格式（简化）**：
```json
{
  "source_file": "fib_student_a.py",
  "functions": [
    {
      "name": "fibonacci",
      "lineno": 1,
      "nodes": [{"nid": 0, "kind": "ASSIGN", "reads": ["n"], "writes": ["a"], ...}],
      "pdg_edges": [{"src": 0, "dst": 1, "kind": "DATA_DEP"}],
      "effect_signature": {"counts": {"FILE_IO": 0, "R_STACK": 3, ...}},
      "tokens_norm": ["LOAD_FAST", "COMPARE_OP", "POP_JUMP", ...]
    }
  ]
}
```

**作用**：将源代码转换为统一的中间表示，后续所有步骤都基于此 JSON 工作，不再依赖原始源代码。

---

### 阶段 2：多视图增强（基础 JSON → 多视图 JSON）

**脚本**：`scripts/enrich_graph_views.py`

```bash
python scripts/enrich_graph_views.py \
  -j data/artifacts/json \
  -o data/artifacts/json_mv
```

**做了什么**：
1. 读取阶段 1 产出的 `*.epdg.json`
2. 从 PDG 边中分离出明确的 DFG（数据流图）边和 AST（语法树）边
3. 输出带有 `dfg_edges`、`ast_edges` 字段的增强 JSON

**作用**：为后续的图指纹计算提供多视图表示。不同视图捕获代码不同维度的结构信息，提高查重的鲁棒性。

---

### 阶段 3：构建 LSH 索引（多视图 JSON → 索引文件）

**脚本**：`scripts/build_index.py`

```bash
python scripts/build_index.py \
  -j data/artifacts/json_mv \
  -o data/artifacts/index/epdg_lsh.pkl
```

**做了什么**：
1. 遍历所有 `*.epdg.json` 文件，提取每个函数
2. **Token 索引**：将 opcode token 序列切成 k-gram → 计算 MinHash 签名 → 插入 LSH 表
3. **Graph 索引**：在 PDG 上运行 WL-hash（Weisfeiler-Lehman）迭代 + k-path 枚举 → 生成图指纹 → 计算 MinHash → 插入 LSH 表
4. **副作用向量**：将副作用签名归一化为单位向量
5. 输出：包含双索引的 pickle 文件

**关键参数**（可通过命令行调整）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--num_perm` | 128 | MinHash 排列数（越大越精确但越慢） |
| `--bands` | 32 | LSH 分段数（影响召回率/精确度平衡） |
| `--shingle_k` | 5 | Token k-gram 长度 |
| `--wl_iters` | 2 | WL-hash 迭代次数 |
| `--kpath_k` | 3 | k-path 路径长度 |

**作用**：构建高效的近似最近邻索引，使得后续查询可以在 O(1) 时间找到相似函数，避免 O(n²) 的全量比对。

---

### 阶段 4：生成候选对（索引 + JSON → 候选函数对）

**脚本**：`scripts/build_lsh_candidates.py`

```bash
python scripts/build_lsh_candidates.py \
  -i data/artifacts/index/epdg_lsh.pkl \
  -j data/artifacts/json_mv \
  -o data/artifacts/retrieval/lsh_pairs.json
```

**做了什么**：
1. 对每个函数，查询 token LSH 和 graph LSH，获取邻居候选
2. 按 token 长度过滤掉过短的函数（避免短小工具函数产生误报）
3. 用副作用向量余弦相似度对候选排序
4. **模板检测**：统计每个函数的邻居度（neighbor degree），度数过高的函数标记为"模板函数"（如通用的 main 函数、打印函数等）
5. **硬过滤**：如果一对函数都是超高频模板，直接剔除（因为大家都写一样的模板不代表抄袭）

**输出格式**：
```json
{
  "pairs": [
    {"lsh_sim": 0.93, "meta_i": {"file": "a.py", "func": "fib", ...}, "meta_j": {...}},
    ...
  ],
  "meta": {"num_funcs": 53, "num_pairs": 1180}
}
```

**作用**：快速筛选出可能相似的函数对，大幅缩小后续精确比较的范围。

### 阶段 5：程序级评分 + HTML 报告

**脚本**：`scripts/search_and_rank.py`

```bash
python scripts/search_and_rank.py \
  -i data/artifacts/index/epdg_lsh.pkl \
  -j data/artifacts/json_mv \
  -o data/artifacts/retrieval/baseline_pairs.json \
  -r data/artifacts/reports/baseline_pairs.html \
  --threshold 0.5 \
  --min_coverage 0.3 \
  --candidates data/artifacts/retrieval/lsh_pairs.json
```

**做了什么**：
1. 加载候选函数对，进行重排序评分
2. **函数级 → 程序级聚合**：将函数对按所属源文件分组
3. **覆盖率计算**：对每个程序对，计算匹配函数占总函数的比例
   - `coverage_A = sum(匹配函数长度) / sum(A的所有函数长度)`
   - `combined_coverage = max(coverage_A, coverage_B)`
4. **最终评分**：`final_score = base_score × combined_coverage`
5. 过滤：仅保留 `final_score ≥ threshold` 且 `coverage ≥ min_coverage` 的对
6. 生成交互式 HTML 报告

**关键参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--threshold` | 0.5 | 程序对相似度阈值，低于此值不报告 |
| `--min_coverage` | 0.3 | 最低覆盖率，确保有足够多的函数匹配 |
| `--dacd_clusters` | 无 | 可选，传入 DACD 聚类结果以降低模板权重 |

**作用**：最终的评分和输出阶段，产生可直接查看的查重报告。

---

### 阶段 6（可选）：DACD 模板聚类

**脚本**：`clustering/hdbscan_cluster.py`（通过 `python -m` 调用）

```bash
python -m clustering.hdbscan_cluster \
  --json-root data/artifacts/json_mv \
  --output data/artifacts/json_mv/dacd_clusters.json \
  --min-cluster-size 8 \
  --epsilon 0.35
```

**做了什么**：
1. 收集所有 JSON 中的函数
2. 计算 DACD 距离矩阵：`dist = w_graph × (1 - graph_jaccard) + w_effect × (1 - cosine_sim)`
3. 运行 HDBSCAN 聚类（失败时降级为 sklearn AgglomerativeClustering）
4. 识别模板簇：成员多、分布在多个不同文件中的簇
5. 为每个函数分配模板权重 `w ∈ (0.1, 0.8]`

**作用**：识别通用模板代码（如标准的 main 函数、常见输入处理），在评分时降低这些函数的贡献权重，减少误报。

## 已知问题与注意事项

### 1. build_epdg 会处理 data/submissions 目录

当使用 `--src_root data/submissions_simple` 时，阶段 1 只会处理 `submissions_simple` 下的文件。但由于之前运行可能在 `data/artifacts/json/` 中留下了 `submissions/` 子目录的 JSON 文件，后续阶段会将这些文件也纳入索引和比较。

**建议**：每次运行新数据前，先清空 artifacts 目录：
```bash
rm -rf data/artifacts/json data/artifacts/json_mv data/artifacts/index data/artifacts/retrieval data/artifacts/reports
```

### 2. C/C++ 支持需要 clang

C/C++ 前端依赖系统安装的 `clang` 编译器。如果只检测 Python 代码，不需要安装。

### 3. Java 支持需要 javac/javap

Java 前端依赖 JDK 工具链。

### 4. datasketch 依赖可能多余

`requirements.txt` 中的 `datasketch>=1.5.9` 实际上在代码中未被 import。系统使用自己实现的 `retrieval/minhash_lsh.py`。可以考虑从 `requirements.txt` 中移除以精简依赖。

### 5. VCoME 相关代码已移除

代码中多处有 VCoME（一个图神经网络编码器）的兼容层和降级逻辑。VCoME 已被移除，系统完全基于结构特征工作，不需要深度学习框架。`requirements.txt` 中深度学习相关的依赖行已清空（第13-14行）。

### 6. 模板函数误报

对于非常短小或非常通用的函数（如只有 `return` 语句的函数），LSH 可能产生较多匹配。`--with_dacd` 模式可以缓解此问题。

## 算法原理简述

1. **E-PDG**：将代码解析为节点（语句/表达式）和边（数据依赖、控制依赖、副作用依赖）的图结构，附加副作用签名（文件I/O、网络、数据库等）
2. **MinHash/LSH**：将图指纹和 token 序列转换为固定长度的哈希签名，通过 LSH 分桶实现 O(1) 近邻查询
3. **WL-hash**：Weisfeiler-Lehman 图同构测试的变体，通过迭代聚合邻居标签来捕获图的局部结构
4. **覆盖率评分**：不仅看单个函数的相似度，还看匹配函数占整个程序的比例，避免因一两个通用函数就判定为抄袭
5. **DACD 模板降权**：自动识别通用模板代码，降低其在评分中的权重
